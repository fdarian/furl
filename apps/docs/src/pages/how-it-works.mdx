# How It Works

`furl` resolves markdown by running an ordered list of **resolvers** implemented in `packages/core`. By default that list is furl's six built-in strategies, tried in this fixed order:

## Fallback chain

1. Extension passthrough

If the URL pathname ends with a file extension, `furl` fetches the body as-is and reports `raw`.

2. Direct markdown request

For non-file URLs, `furl` sends a `GET` with `Accept: text/markdown`. If the response `content-type` contains `markdown`, it returns the body and reports `direct`.

3. `.md` suffix retry

If the direct fetch does not return markdown, `furl` retries the same URL with `.md` appended before any query string. A successful markdown response reports `md-suffix`.

4. Provider fallback

If all three probes above decline, `furl` falls through to the provider builtins, in order:

- `jina`
- `exa`
- `firecrawl`

## Source labels

The CLI writes the markdown body to stdout and a one-line diagnostic to stderr, naming whichever resolver produced the markdown:

```text
↳ via raw
↳ via direct
↳ via md-suffix
↳ via jina
↳ via exa
↳ via firecrawl
```

A plugin that resolves the URL reports its own manifest `name` instead (e.g. `↳ via x`).

## Default behavior

`jina` is the first provider builtin tried, and it works without an API key by routing requests through `https://r.jina.ai/<url>`.

## Plugins

This fixed chain is exactly what the plugin system extends. Each stage above is a `default:` built-in (`default:raw`, `default:direct`, `default:md-suffix`, `default:jina`/`default:exa`/`default:firecrawl`, detailed in [Builtins](/plugins/builtins/fetch-strategies)) that can be named in the `order` config; plugins run ahead of them to handle sites the generic chain can't, like client-rendered pages. See [Plugins](/plugins/overview) for the resolution algorithm.
